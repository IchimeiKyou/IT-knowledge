1コンピューターハードウェア入門

💬 Momoとめいめいの会話 — PCの電源を入れただけでなぜ動くの？
🌸 Momo PCって電源ボタンを押すだけでYouTubeが見られるじゃないですか。中で何が起きてるのか全然想像できないんですよね……
💻 めいめい PCの中で一番大事なのは3つのパーツです。まず「CPU」は人間で言うと「脳みそ」——計算を全部担当します。次に「RAM（メモリ）」は「作業机」——今使っているデータを広げておく場所。最後に「ストレージ（HDD/SSD）」は「本棚」——電源を切っても消えないデータの保管場所です。
🌸 Momo 「メモリが足りない」とか「SSDが速い」って聞くんですが、何が変わるんですか？
💻 めいめい メモリが少ないと「作業机が狭くて書類があふれる」状態になり、PCがもたつきます。SSDはHDDの10倍以上速くデータを読み書きできるので、Windowsの起動が「2分→15秒」に縮まります。プログラムの動作速度も、CPUが速いかどうか・メモリが十分あるかどうかに大きく左右されます。
🌸 Momo PCスペックを見るとき「Core i7・16GB・512GB SSD」って書いてあるのって、CPU・RAM・ストレージのことなんですね！
💻 めいめい 完璧な理解です！エンジニアとして仕事をするとき、「このサーバーのCPUとメモリが足りているか」「ストレージの読み書き速度がボトルネックではないか」を判断するシーンが頻繁にあります。ハードウェアの基礎は必須知識ですよ。

CPU（中央処理装置）

CPUCPU: Central Processing Unit（中央処理装置）
正式定義コンピューターの中核となる演算処理装置。算術論理演算ユニット（ALU）・制御ユニット（CU）・レジスタから構成され、プログラムの命令を取得（Fetch）→解読（Decode）→実行（Execute）→書き戻し（Write Back）のサイクルで処理する。性能指標はクロック周波数（GHz）・コア数・キャッシュサイズ。
わかりやすくコンピューターの「脳みそ」。計算・比較・分岐（if/else）・メモリ操作などすべての処理を担う。Core i9・Ryzen 9など高性能CPUは同時に多くの計算を並列実行できる（多コア）。
主要メーカー・製品Intel（Core i3/i5/i7/i9シリーズ）、AMD（Ryzenシリーズ）、Apple（M1/M2/M3、ARMアーキテクチャ）、サーバー向けはIntel Xeon・AMD EPYC

メモリ（RAM）とストレージ

RAM（メインメモリ）RAM: Random Access Memory
正式定義CPUが現在処理中のデータとプログラムを一時保存する揮発性記憶装置（Volatile Memory）。電源を切るとデータが消える。アクセス速度はストレージの数十〜数百倍。容量単位はGB（ギガバイト）で表し、一般的なPCは8〜32GB。
わかりやすく「作業机の広さ」。Chromeのタブを100個開くと「机がいっぱいで新しい書類を置けない」状態（メモリ不足）になる。開発用PCは最低16GB、推奨32GBが多い。
ストレージ（HDD / SSD）Storage: HDD / SSD
正式定義電源を切っても消えない不揮発性の大容量記憶装置。HDD（Hard Disk Drive）は磁気ディスク・回転式で大容量・低価格。SSD（Solid State Drive）はフラッシュメモリ使用・高速・耐衝撃性高・高価格。サーバーの高速ストレージにはNVMe SSDが使われる。
わかりやすく「本棚・倉庫」。電源を切っても消えないため、OSやアプリ・写真・動画を保存する。SSDはHDDと比べて起動やアプリ起動が約10倍速い。

入力・出力デバイス

入力デバイス / 出力デバイスInput Device / Output Device
正式定義入力デバイスはユーザーからコンピューターへ情報を送る機器（キーボード・マウス・タッチパネル・マイク・カメラ）。出力デバイスはコンピューターからユーザーへ情報を伝える機器（ディスプレイ・スピーカー・プリンター）。USB・HDMI・Bluetooth等の規格で接続される。
わかりやすく「人間 → PC」に情報を送るのが入力、「PC → 人間」に情報を届けるのが出力。キーボード入力 → CPUが処理 → ディスプレイに表示、という流れが基本。
代表的な入力デバイスキーボード・マウス・タッチスクリーン・Webカメラ・マイク・スキャナー・ゲームコントローラー
パーツ役割の例え性能指標容量/速度の目安（一般PC）
CPU脳みそ・計算担当クロック周波数（GHz）・コア数4コア〜16コア、2〜5GHz
RAM作業机の広さ容量（GB）・転送速度（MHz）8〜64GB（開発用は16GB以上推奨）
SSD高速本棚容量（GB/TB）・読み書き速度（MB/s）256GB〜2TB、読み書き3,000MB/s（NVMe）
HDD大容量倉庫容量（TB）・回転数（RPM）1〜8TB、読み書き100〜200MB/s
GPU映像・AI専用計算機VRAMサイズ・CUDA/テンソルコア数4〜24GB VRAM（AI用は80GB+）
マザーボード全パーツをつなぐ基盤チップセット・PCIeスロット数CPU・RAM・SSDをすべて搭載
💡 エンジニアとしてハードウェアを学ぶ理由：サーバーの性能チューニング・スペック選定・パフォーマンスボトルネック調査など、インフラ・バックエンド・DevOps 業務でハードウェアの基礎知識は日常的に必要になります。「なぜこのサーバーは遅いのか」を判断するには CPU 使用率・メモリ使用量・ディスク I/O の3点を確認するのが基本です。
CPUとメモリの違いを教えてください
「CPUは処理をして、メモリはデータを保存するものだと思います」
「CPUは算術論理演算ユニットと制御ユニットを持つ演算処理装置で、プログラムの命令を実行します。RAM（メインメモリ）はCPUが処理中のデータを一時保存する揮発性記憶装置で、電源を切ると消えます。CPUが"計算を行う脳"、RAMが"その脳が今使っているデータを広げる机"の関係です。ストレージ（SSD/HDD）は電源を切っても消えない長期保存領域で、机の引き出しや本棚にあたります。」
💬 CPUとは何ですか？
友人説明パソコンの脳みそ。計算とか命令の処理を全部やってる部品だよ。
面接基本CPUはコンピューターの中央処理装置で、プログラムの命令を順番に解釈・実行する役割を担っています。
面接＋αCPUはフェッチ・デコード・実行・ライトバックの4ステージで命令を処理し、クロック周波数とコア数が性能を決定します。
模範回答CPUは命令セットアーキテクチャに基づきパイプライン処理を行います。マルチコア化によりスレッドレベル並列性を活かし、キャッシュ階層（L1/L2/L3）でメモリレイテンシを隠蔽しています。
💬 RAMとストレージの違いを説明してください
友人説明RAMは作業机で、SSDは引き出し。電源切ったらRAMの内容は消えるけどSSDは残るよ。
面接基本RAMは揮発性の一時記憶領域でCPUが直接アクセスし、ストレージは電源オフでも保持される永続記憶装置です。
面接＋αRAMはDRAMベースで容量GBオーダー・レイテンシns単位、SSDはフラッシュメモリでTBオーダー・レイテンシμs単位。用途・コスト・速度のトレードオフです。
模範回答アクセス速度はRAM（数十ns）がNVMe SSD（数十μs）より約1000倍高速です。OSはページングにより仮想アドレス空間をRAMとスワップ領域に透過的にマッピングし、メモリ階層全体の有効利用を実現しています。

2IT業界の全体像

💬 Momoとめいめいの会話 — SIerと自社開発、どっちがいいの？
🌸 Momo 就活サイトに「SIer」と「自社開発」ってよく出てくるんですけど、何が違うんですか？
💻 めいめい 分かりやすく言うと、SIerは「他の会社のシステムを受注して作る」会社。自社開発は「自分の会社のサービスを自分たちで作り続ける」会社です。コンビニで例えると、SIerは「フランチャイズ本部から依頼された改装工事をする業者」で、自社開発は「自分たちで新メニューを開発してお客さんに届けるチェーン」みたいな感じですね。
🌸 Momo なるほど！じゃあSIerは色々な業種のシステムに関われそうですね。
💻 めいめい そうです。銀行・病院・行政など幅広い業界のシステムを担当できるのがSIerの醍醐味。一方、自社開発はECサービスやアプリなど特定のプロダクトを深く作り込めるのが魅力です。面接では「なぜその働き方を選んだか」をしっかり言語化しておくといいですよ。

2-1 IT業界の主要セクター

日本のIT業界は大きく4つのセクターに分類されます。志望企業がどのセクターかによって、面接で問われる内容も異なります。

SIer（エスアイアー）SIer: System Integrator
正式定義顧客企業からシステム開発・構築・運用を一括して受託し、要件定義から設計・開発・テスト・保守までのライフサイクル全体を管理するIT企業。請負契約または準委任契約に基づき、プロジェクト単位で業務を遂行する。
わかりやすく「お客さんの会社に代わって、システムをまるごと作ってあげる会社」。例えば銀行の振込システムや、大手流通の在庫管理システムを、銀行・流通会社の代わりに丸ごと請け負って作る。
具体例富士通がみずほ銀行の勘定系システムを受注 → 要件ヒアリング → 設計書作成 → 開発（プログラミング）→ テスト → 納品 → 保守。このプロジェクトが「SI（System Integration）」案件。

2-2 職種の全体像

💼 SE（Systems Engineer）

要件定義・設計・実装・テストを担当。技術的な中核職種。未経験からでも入社後にキャリアを積める。

📊 PM（Project Manager）

スケジュール・予算・品質・人員を管理。技術力よりコミュニケーション・調整能力が重視される。

💡 ITコンサルタント

経営課題をIT視点で解決提案。MBAや論理思考が重視。マッキンゼー出身者も多い。

🛒 ITセールス/プリセールス

IT製品・ソリューションを企業に提案・販売。技術理解＋営業力が必要。対人スキルが活きる職種。

🧪 QAエンジニア（Quality Assurance）

ソフトウェアの品質を保証するテスト設計・実行・自動化を担当。バグを防ぐ守りの要。テスト設計力と論理的思考が核心。

📈 データサイエンティスト

統計・機械学習でビジネス課題を解決する。データ分析・可視化・モデル構築が業務の中心。数学・統計の素養が必要。

⚙️ SRE（Site Reliability Engineer）

Googleが提唱したサイト信頼性エンジニアリング。可用性・レイテンシ・パフォーマンスをソフトウェアエンジニアリングで向上させる。DevOpsとの深い関連。

🔒 セキュリティエンジニア

システムの脆弱性診断・セキュリティ設計・インシデント対応を担当。需要は急増中。倫理観と技術力の両方が求められる。

💬 SIerとはどんな会社ですか？
友人説明他の会社のシステムを受注して作ってあげる会社だよ。大手銀行とか官公庁のシステム担当が多い。

面接＋α大手SIerは多重下請け構造でプロジェクト管理・設計を担い、開発工程を協力会社に委託します。近年はクラウド対応やDX推進でコンサル機能も強化しています。
模範回答SIerはウォーターフォール型開発が主流でしたが、アジャイル・DevOps導入が加速しています。私はSIerと自社開発双方のビジネスモデルを理解した上で、貴社の◯◯という強みに魅力を感じ志望しました。
💬 受託開発と自社開発の違いを教えてください
友人説明受託は他社のシステムを作るお仕事、自社開発は自分たちのサービスを自分たちで作る感じ。
面接基本受託開発はクライアントの要件に従いシステムを開発する形態、自社開発は自社サービス・プロダクトを継続的に開発・改善する形態です。
面接＋α受託は納品形式でコスト管理が重要、自社開発はKPI・ユーザー価値最大化を直接追求できます。キャリアパスや技術スタック選択の自由度も大きく異なります。
模範回答受託開発では多様な業界知識と折衝力が培われ、自社開発ではプロダクト思考とデータドリブン改善が強化されます。貴社は自社プロダクト開発を軸としており、私の◯◯経験を直接ユーザー価値に転換できると考えています。

3インターネット・ネットワーク基礎

💬 Momoとめいめいの会話 — ブラウザにURLを入力したら何が起きるの？
🌸 Momo 「google.com」ってブラウザに打ったとき、瞬時にページが出てきますよね。裏で一体何が起きてるんですか？
💻 めいめい 手紙の配達に例えると分かりやすいです。まず「google.com」という名前を住所に変換する作業があります。これがDNS（ドメインネームシステム）。「google.com＝142.250.xxx.xxx」というIPアドレスに変換してくれる、インターネット版の住所録です。
🌸 Momo 住所録みたいな存在なんですね！そのあとは？
💻 めいめい 次にTCPで「つながりました！」という3回の確認（3ウェイハンドシェイク）を行い、それからHTTPSで「このページください」とリクエストを送ります。サーバーが「はい、どうぞ」とHTMLを返して、ブラウザが画面に描画する——この全部が0.2秒以内に終わってるんですよ。
🌸 Momo すごい……！URL1回押しただけなのに、そんなにたくさん通信してるんですね。

3-1 クライアント・サーバーモデル

クライアント・サーバーモデルClient-Server Model
正式定義ネットワーク上でサービスを要求する側（クライアント）と、サービスを提供する側（サーバー）に役割を分離したアーキテクチャ。クライアントはリクエストを送信し、サーバーはそれを受信・処理してレスポンスを返す非対称な通信モデル。
わかりやすくお客（クライアント）がレストランに注文（リクエスト）を出し、厨房（サーバー）が料理（レスポンス）を返す構造。ブラウザでURLを入力すると、あなたのPCがクライアントになり、Webサーバーにページを要求する。
具体例Googleにアクセス: ①ブラウザがDNSでIPアドレスを解決 → ②HTTPリクエストをGoogleサーバーに送信 → ③Googleサーバーが検索結果HTMLを返す → ④ブラウザがHTMLを描画

3-2 OSI 7層モデル（OSI Reference Model）

OSI参照モデルOSI: Open Systems Interconnection Reference Model
正式定義ISOが策定した、ネットワーク通信を機能ごとに7つの階層（レイヤー）に分割した標準的な概念モデル。各層は隣接する層にのみサービスを提供・利用し、上位層が下位層のサービスを使用する。実際のインターネットはTCP/IPモデル（4層）が使われるが、OSIモデルは通信プロトコルの教育・設計の共通語として広く活用される。
わかりやすく荷物を国際郵便で送る際の分業体制に相当。梱包→住所記入→配送車→飛行機→現地配送→受取という各役割を分担。ネットワーク障害時に「L3の問題です」と言えば、IPアドレス設定ミスが疑われ、「L2の問題です」ならLANケーブル/スイッチを確認する。

TCP/IP 5層モデル（インターネットの実用アーキテクチャ）

OSI vs TCP/IP：OSI参照モデルは「理論的な教科書モデル」として通信プロトコルの理解・設計に使われます。しかし実際のインターネットが動いているのはTCP/IPモデルです。エンジニアが「L3の問題（ルーティング）」「L4レベルの話（TCP接続）」と言う場合、ほぼ必ずこのTCP/IPの層を指しています。OSIは概念理解の地図、TCP/IPは実際の道路と覚えましょう。
TCP/IPモデル（インターネットモデル）TCP/IP 5-Layer Reference Model
正式定義インターネットの実装に対応した階層モデル。RFC 1122等で規定され、アプリケーション・トランスポート・インターネット・データリンク・物理の5層に分類される。OSIの7層を実用的に統合したモデルであり、実際のプロトコルスタック（TCP/IP）の実装に直接対応している。HTTPSで通信するとき、データはアプリ層で生まれ→TCP層でセグメント化→IP層でパケット化→データリンク層でフレーム化→物理層で電気/光信号として伝わる。
わかりやすく宅配便の仕組みに例えると：荷物（HTTPデータ）を梱包し（TCP）→住所ラベルを貼り（IP）→配送トラックに載せ（Ethernet）→道路を走る（物理）——各層が役割を分担して目的地まで届ける。

3-3 HTTP・HTTPS とステータスコード

HTTP / HTTPSHTTP: HyperText Transfer Protocol / HTTPS: HTTP Secure
正式定義HTTPはWebブラウザとWebサーバー間でHTMLを転送するための、ステートレスなアプリケーション層プロトコル（RFC 7230〜7235）。HTTPSはHTTPにTLS（Transport Layer Security）による暗号化・認証・完全性保護を追加したプロトコル。標準ポートはHTTPが80番、HTTPSが443番。
わかりやすくHTTPは「Webページを取得する通信ルール」。HTTPSはその通信内容を暗号化したバージョン。ネットショッピングや銀行サイトのURLに「https://」と鍵マーク🔒が表示されるのがHTTPS接続の証拠。
具体例HTTP通信の基本形式: クライアントが「GET /index.html HTTP/1.1」を送信 → サーバーが「HTTP/1.1 200 OK ＋ HTMLコンテンツ」を返す。

3-4 DNS（Domain Name System）

DNSDNS: Domain Name System
正式定義インターネット上のホスト名（ドメイン名）とIPアドレスを相互変換する分散型データベースシステム（RFC 1034/1035）。階層的な名前空間構造を持ち、ルートDNSサーバーから権威DNSサーバーまで再帰的に問い合わせを行う。
わかりやすくインターネットの電話帳。人間が覚えやすい「google.com」という名前を、コンピューターが理解できるIPアドレス「142.250.196.46」に変換する仕組み。
具体例ブラウザに「amazon.co.jp」を入力 → DNSに問い合わせ → 「52.119.168.1」のIPアドレスが返る → そのIPのサーバーに接続。

3-5 API（Application Programming Interface）

APIAPI: Application Programming Interface
正式定義ソフトウェアコンポーネントが互いに通信するための仕様・規約の集合体。特にWeb APIにおけるREST（Representational State Transfer）は、HTTPプロトコルを使用し、リソースをURLで表現し、HTTPメソッド（GET/POST/PUT/DELETE）でCRUD操作を行う設計原則に基づく。
わかりやすく異なるアプリ同士が会話するための規格化された窓口。天気アプリがスマホメーカーと気象庁という全く別の会社のデータを繋いで表示できるのはAPIのおかげ。
具体例「Google Maps API」をECサイトに組み込むと、ECサイト上にGoogleマップが表示できる。

APIとは何か説明してください
アプリ同士をつなぐものだと思います。
APIとは、異なるソフトウェア間で機能やデータをやり取りするための仕様・規約の集合体です。Web APIではHTTPプロトコルを介してリクエストを送ると、JSONやXML形式でデータが返ります。例えば地図アプリがGoogle Maps APIを呼び出すことで、Googleのサーバーから地図データを取得して自社アプリ内に表示できます。

3-6 TCP と UDP（トランスポート層プロトコル）

TCP / UDPTCP: Transmission Control Protocol / UDP: User Datagram Protocol
正式定義TCPはRFC 793で定義された、コネクション指向・信頼性保証型のトランスポート層プロトコル。3ウェイハンドシェイク（SYN→SYN-ACK→ACK）で接続を確立し、データの到達確認・順序保証・フロー制御・輻輳制御を行う。UDPはRFC 768で定義されたコネクションレス型のプロトコルで、ヘッダーオーバーヘッドが最小限であり、信頼性より低遅延・高スループットを優先する用途に使用される。
わかりやすくTCPは「確実に届ける宅配便」（受取確認あり）、UDPは「投げっぱなし郵便」（受取確認なし）。Webページ閲覧やメールはデータの欠損が許されないのでTCP。動画会議や音声通話は少々欠けても速さが大事なのでUDPが向く。

速度ヘッダーが大きく遅延ありヘッダー最小、超低遅延

TCP/IPとは何ですか？HTTP通信との関係を教えてください
インターネットで使う通信プロトコルです。
TCP/IPは、インターネット通信の基盤となるプロトコル群の総称です。IPがデータをどのアドレスに届けるか（経路）を担当し、TCPが届いたかどうかの確認・順序保証を担当します。HTTPはその上位層で動く「何を・どう要求するか」の規約で、TCPを使ってデータの確実な転送を保証しながら動作します。つまり「HTTP over TCP over IP」という階層構造になっています。
HTTPポート80番とHTTPSポート443番の違いは何ですか？
HTTPSのほうが安全です。ポート番号が違います。
HTTPは80番ポートを使い、通信内容が暗号化されていないため第三者に盗聴されるリスクがあります。HTTPSは443番ポートを使い、TLS（Transport Layer Security）による暗号化で通信内容を保護します。現代のWebサービスはHTTPSが標準で、Googleはhttps非対応サイトのSEO評価を下げるほか、ChromeブラウザもHTTPサイトを「保護されていない通信」と警告します。本番環境では原則HTTPSのみ許可するのが鉄則です。
💬 HTTPとHTTPSの違いを説明してください
友人説明HTTPは通信が丸見えで、HTTPSは暗号化されてて安全。南京錠マークがついてる方がHTTPS。
面接基本HTTPSはHTTPにTLS暗号化を加えたプロトコルです。通信内容を暗号化し、サーバー証明書による認証で盗聴・改ざんを防ぎます。
面接＋αTLSハンドシェイクで公開鍵交換・セッション鍵生成を行い、AES等の対称暗号で実データを保護します。HSTS設定でHTTPへのダウングレード攻撃も防止できます。
模範回答TLS 1.3ではハンドシェイクが1-RTTに短縮され、前方秘匿性（PFS）をECDHEで保証します。証明書はルートCA→中間CA→サーバー証明書のチェーンで信頼性を検証し、OCSP staplingで失効確認を効率化しています。
💬 DNSとは何ですか？仕組みを説明してください
友人説明ドメイン名（google.com）をIPアドレスに変換する電話帳みたいなシステム。
面接基本DNSはドメイン名をIPアドレスに変換する分散型名前解決システムです。ブラウザはDNSサーバーへ問い合わせ、取得したIPでサーバーへ接続します。
面接＋α解決はルートDNS→TLD DNS→権威DNSの階層クエリで行われます。TTL設定でキャッシュ期間を制御し、ネガティブキャッシュでNXDOMAIN応答も効率化しています。
模範回答再帰リゾルバはスタブリゾルバから受けたクエリをフルサービス解決します。DNSSEC署名でキャッシュポイズニングを防止し、DNS over HTTPS（DoH）で通信プライバシーを保護するのが現代のベストプラクティスです。

4-1 プログラミング言語とは

わかりやすくコンピューターはゼロとイチしか理解できない。それを人間が読める言葉で書けるようにしたのがプログラミング言語。英語の指示書をコンピューター語（機械語）に翻訳する作業が「コンパイル」。

4-2b Hello World で言語を比べてみよう

💡 なぜ Hello World を比べると言語の個性がわかるの？
プログラマーが新しい言語を学ぶとき最初に書くのが「Hello, World!」。たった1〜数行なのに、その構文に言語の設計思想（静的 vs 動的型付け・コンパイル型 vs スクリプト型・オブジェクト指向 vs 手続き型）が凝縮されています。
# Python — シンプル重視、1行でOK print("Hello, World!")
// JavaScript — ブラウザ・Node.js 両対応 console.log("Hello, World!");
// TypeScript — 型注釈が追加される const msg: string = "Hello, World!"; console.log(msg);
// Java — クラスと main メソッドが必須 public class Main { public static void main(String[] args) { System.out.println("Hello, World!"); } }
# Ruby — 人間に優しいシンタックス puts "Hello, World!"
// Go — package と import が必須 package main import "fmt" func main() { fmt.Println("Hello, World!") }
設計思想の違い：動的型付け vs 静的型付け
Python・Ruby・JavaScriptは変数の型を書かなくてよい動的型付け言語（柔軟・初心者に親しみやすい）。TypeScript・Java・Goは型を明示する静的型付け言語（大規模開発でバグを早期発見しやすい）。Javaは1行出力するだけでもクラス定義が必要なほど型と構造を重視しており、エンタープライズ開発の堅牢性を体現しています。

4-5 フロントエンドとバックエンド（Frontend vs Backend）

フロントエンドFrontend (Client-Side)
正式定義ユーザーのWebブラウザやアプリ上で直接実行される表示層の総称。HTML（文書構造）・CSS（スタイリング）・JavaScript（インタラクション）で構成される。ブラウザのレンダリングエンジンが解釈・実行する。
わかりやすくユーザーが目で見て触れる部分すべて。ボタン・テキスト・画像・アニメーション・フォームなどがフロントエンドです。「見た目を作る仕事」とも言えます。
主要技術HTML/CSS/JavaScript・React・Vue.js・Angular・TypeScript・Tailwind CSS
バックエンドBackend (Server-Side)
正式定義Webサーバーまたはアプリケーションサーバー上で実行される処理層の総称。ビジネスロジック・データベースアクセス・認証・APIの実装などを担う。クライアントからは直接見えない。
わかりやすくユーザーには見えない「裏側の仕組み」。「Aさんの注文をDBに保存する」「ログイン認証する」「支払いを処理する」などの処理はバックエンドで行います。
主要技術Node.js・Python（Django/FastAPI）・Java（Spring Boot）・Ruby（Rails）・Go・PHP（Laravel）・MySQL/PostgreSQL
面接頻出：フルスタックエンジニア（Full-Stack Engineer）とは？
フロントエンドとバックエンドの両方を担当できるエンジニアを指します。スタートアップや少人数チームで重宝されます。大手SIerではフロント・バックを専門分業するケースも多い。TypeScript + Node.js + React が一言語（JS）で統一できるため人気があります。
フロントエンドとバックエンドの違いを説明してください
フロントエンドは見た目で、バックエンドは裏側の処理です。
フロントエンドはユーザーのブラウザ上で実行されるHTML・CSS・JavaScriptによる表示層です。ReactやVue.jsで動的なUIを構築します。バックエンドはサーバー側で動作し、ビジネスロジック・データベースアクセス・認証・REST APIの提供を担います。両者はHTTPを介したAPIで通信し、フロントエンドがJSONデータを受け取り表示します。両方担当できるのがフルスタックエンジニアです。

💬 O記法（ビッグオー）で計算量を説明してください
友人説明データが増えた時にどのくらい処理が遅くなるかを表す指標。O(n²)は最悪、O(log n)は優秀みたいな感じ。

6-1 RDBMSとは

RDBMS（リレーショナルデータベース）RDBMS: Relational Database Management System

わかりやすくExcelの表が複数あって、それぞれの表が「外部キー」というリンクで繋がったデータ管理システム。MySQLやPostgreSQLが代表例。

6-2 SQL（Structured Query Language）の基本操作

-- テーブル定義（DDL: Data Definition Language） CREATE TABLE employees ( employee_id INT PRIMARY KEY AUTO_INCREMENT, name VARCHAR(100) NOT NULL, department_id INT REFERENCES departments(id), salary DECIMAL(10,2), hire_date DATE NOT NULL ); -- データ挿入（DML: Data Manipulation Language） INSERT INTO employees (name, department_id, salary, hire_date) VALUES ("山田太郎", 3, 450000, "2020-04-01"); -- データ検索（SELECT） SELECT e.name, d.dept_name, e.salary FROM employees e INNER JOIN departments d ON e.department_id = d.id WHERE e.salary > 400000 ORDER BY e.salary DESC LIMIT 10; -- データ更新・削除 UPDATE employees SET salary = 500000 WHERE employee_id = 1; DELETE FROM employees WHERE employee_id = 99;

6-3 ER図（Entity-Relationship Diagram）

ER図ERD: Entity-Relationship Diagram
正式定義データベース設計において、エンティティ（実体）、属性（Attribute）、リレーションシップ（関係）をグラフィカルに表現した設計図。エンティティ間の多重度（1対1、1対多、多対多）を記述し、テーブル設計の基礎となる。
わかりやすく「DBのテーブルの関係図」。どのテーブルとどのテーブルが繋がっているか、1つの注文に複数の商品が紐づく（1対多）関係などを視覚化する。
CRUDCRUD: Create, Read, Update, Delete
正式定義データベース操作およびREST API設計における4つの基本操作の頭字語。Create（作成）: INSERT / HTTP POST、Read（読み取り）: SELECT / HTTP GET、Update（更新）: UPDATE / HTTP PUT・PATCH、Delete（削除）: DELETE / HTTP DELETE に対応する。
わかりやすくあらゆるデータ操作は「作る・読む・更新する・消す」の4つに集約できます。SNSで言えば：投稿作成（Create）・タイムライン表示（Read）・投稿編集（Update）・投稿削除（Delete）がCRUDに相当します。
SQL/HTTP対応CREATE→INSERT・POST / READ→SELECT・GET / UPDATE→UPDATE・PUT/PATCH / DELETE→DELETE・DELETE

6-4 ACID特性（トランザクション管理）

ACID特性ACID: Atomicity, Consistency, Isolation, Durability
正式定義データベーストランザクション（Transaction）が保証すべき4つの特性の頭字語。Atomicity（原子性）: トランザクション内の全操作が全て完了するか全て失敗するか。Consistency（一貫性）: トランザクション前後でデータの整合性が保たれる。Isolation（独立性）: 並行するトランザクションが互いに影響しない。Durability（永続性）: コミット済みデータは障害後も失われない。
わかりやすく銀行振込の例：Aさんの口座から1万円引いてBさんに入金する処理が途中でシステム障害しても、「引いただけで入金されない」事態にならない仕組みがACID。原子性で「全か無か」、耐久性でコミット後のデータは消えない。
よくある誤解：NoSQL（MongoDB等）はACIDを完全保証しないものが多い。スケーラビリティ優先の設計思想から「結果整合性（Eventual Consistency）」を採用する。面接で「NoSQLとRDBMSの使い分け」を問われたらこの点を必ず触れること。

6-5 インデックス（Index）

データベースインデックスDatabase Index
正式定義データベースのクエリ性能を向上させるためのデータ構造。指定した列の値をB木（B-Tree）やハッシュなどの補助データ構造で管理し、テーブル全行スキャン（フルテーブルスキャン）を回避してO(log n)での高速検索を実現する。PRIMARY KEYには自動的にインデックスが設定される。
わかりやすく本の「巻末索引」と同じ仕組み。索引なしでは「佐藤さんを探せ」と言われたらページを1枚ずつ全部めくる羽目になる（フルスキャン）。索引があれば「さ行→佐藤→203ページ」と一瞬で見つかる。行数が増えるほどインデックスの効果が絶大。
注意点インデックスは検索を速くする一方で、INSERT/UPDATE/DELETE時に索引の更新も必要になるため書き込みが若干遅くなる。検索頻度の高い列（WHERE句・JOIN条件・ORDER BY）に貼るのが基本。
-- インデックス作成の例 -- emailでのログイン検索が多い場合はemail列にインデックスを貼る CREATE INDEX idx_users_email ON users(email); -- 複合インデックス（複数列を組み合わせたインデックス） CREATE INDEX idx_orders_user_date ON orders(user_id, created_at); -- インデックスの有無でクエリ実行計画を確認する（EXPLAIN） EXPLAIN SELECT * FROM users WHERE email = 'yamada@example.com';

RDBMSとNoSQLの違いと使い分けを説明してください。
「NoSQLは新しくて速く、RDBMSは古くて遅いです」
「RDBMSは表形式のデータ構造とACID特性による強い整合性保証が強みで、金融・ERP・ECなど整合性が最優先のシステムに向いています。NoSQLはスキーマが柔軟で水平スケールが得意で、SNSのタイムラインやキャッシュ、IoTのログ収集など大量の非構造化データや超低レイテンシが求められる場面で使います。実務では両方を目的別に組み合わせる（PolyglotPersistence）ケースも多いです。」
データベースのインデックスとは何ですか？なぜ重要なのですか？
「検索を速くするものだと思います」
「インデックスはB木などの補助データ構造を用いてクエリ検索を高速化する仕組みです。インデックスなしでは100万行のテーブル全行を走査（フルスキャン）しますが、インデックスがあればO(log n)で目的の行を発見できます。一方で書き込み時（INSERT/UPDATE/DELETE）にはインデックスの更新コストが発生するため、WHERE句・JOIN・ORDER BYで頻繁に使われる列に絞って設定するのが基本です。EXPLAIN文でクエリの実行計画を確認しインデックスが使われているかチェックできます。」
N+1問題とは何ですか？どう解決しますか？
「データを何度も取得してしまう問題だと思います」
「N+1問題とは、1回のリスト取得クエリ（1回）の後に、各レコードの関連データを個別にN回取得するクエリが発行されてしまうパフォーマンス問題です。例えば注文リスト100件を取得し、各注文のユーザー名を1件ずつSELECTすると合計101回のSQLが実行されます。解決策はJOINでまとめて取得するか、ORMのEager Loading（Ruby on Railsの.includes、LaravelのwithなどのN+1回避機能）を使うことです。」

💬 ACID特性とは何ですか？

7-3 コンテナ技術（Container / Docker）

コンテナ / DockerContainer / Docker

IaaS・PaaS・SaaSの違いを説明してください

7-5 サーバーレスアーキテクチャ（Serverless / FaaS）

サーバーレス / FaaSServerless Architecture / FaaS: Function as a Service
正式定義サーバーのプロビジョニング・管理をクラウドプロバイダーに完全委譲し、開発者がビジネスロジックのコードのみに集中できるアーキテクチャスタイル。FaaS（Function as a Service）はその実装形態で、ステートレスな関数単位でコードをデプロイし、イベント駆動で自動スケーリングするサービスモデル。代表：AWS Lambda、Google Cloud Functions、Azure Functions。
わかりやすく「サーバーを持たない」のではなく、「サーバーの管理をしなくていい」という意味。コードだけ書いて預けると、クラウドがサーバーの起動・スケール・停止を全部やってくれる。アクセスがないときはコストゼロ。
具体例ECサイトの画像リサイズ処理：S3に画像がアップロードされるたびLambda関数が自動起動→サムネイル生成→S3に保存。サーバーを常時起動する必要なし。

管理コストOS・ミドルウェア・セキュリティパッチを自社管理コードのみ管理、インフラはクラウドに委譲
スケーリング手動またはAuto Scaling設定が必要リクエスト数に応じて自動スケール（ゼロから数万まで）
コスト体系起動中は常時課金（使わなくても料金発生）実行時間×実行回数の従量課金（アイドル時コストゼロ）

8-1 CIAの三要素

情報セキュリティのCIA三要素CIA Triad: Confidentiality, Integrity, Availability
正式定義ISO/IEC 27001で定義された情報セキュリティの三大特性。機密性（Confidentiality）: 権限のある者だけが情報にアクセスできること。完全性（Integrity）: 情報が正確で改ざんされていないこと。可用性（Availability）: 必要な時に情報・システムにアクセスできること。この三要素のバランスを保つことが情報セキュリティ管理の核心。
わかりやすく機密性は「見られてはいけない人には見せない」、完全性は「データが勝手に書き換えられていない」、可用性は「必要な時にちゃんと使える」こと。例えばDDoS攻撃は可用性を破壊する攻撃。

8-3 暗号化とTLS/SSL

TLS（Transport Layer Security）TLS: Transport Layer Security / SSL: Secure Sockets Layer
正式定義トランスポート層とアプリケーション層の間に位置し、通信の機密性（対称暗号）・完全性（HMAC）・認証（X.509証明書）を提供するプロトコル。SSLはTLSの前身でありセキュリティ上の脆弱性から非推奨。現在はTLS 1.3が最新標準。HTTPSはHTTPにTLSを組み合わせたもの。
わかりやすくインターネット通信を暗号化するプロトコル。公開鍵暗号方式で鍵を安全に交換し、その後は高速な共通鍵暗号で通信内容を暗号化する。Webサイトの鍵マーク🔒がTLS対応の証拠。

8-4 多要素認証（MFA）

多要素認証MFA: Multi-Factor Authentication
正式定義認証の3要素（知識要素・所持要素・生体要素）のうち、2つ以上を組み合わせた認証方式。知識要素（Knowledge）: パスワード・PIN。所持要素（Possession）: スマートフォン・ハードウェアトークン。生体要素（Inherence）: 指紋・顔認証。2要素のみの場合は2FA（Two-Factor Authentication）と呼ぶ。
わかりやすくパスワードだけでなく、スマホに届くSMSコードも入力しないとログインできない仕組み。パスワードが漏れても第二要素がないとログインできないため安全性が格段に向上する。
情報セキュリティのCIAとは何か説明してください
機密性・完全性・可用性の3つです。
CIAとはISO/IEC 27001で定義された情報セキュリティの三要素です。機密性（Confidentiality）は権限のある者のみ情報にアクセスできること、完全性（Integrity）はデータが正確で改ざんされていないこと、可用性（Availability）は必要な時に確実にシステムにアクセスできることを指します。例えばランサムウェアはデータを暗号化するため可用性と機密性を同時に脅かし、DDoS攻撃は可用性を直接破壊します。

8-5 認証と認可・JWT/OAuth2（Authentication vs Authorization）

認証（AuthN）vs 認可（AuthZ）Authentication vs Authorization
正式定義認証（Authentication / AuthN）: システムが「あなたは誰か？」を確認するプロセス。ユーザーの身元・正当性を検証する（パスワード・バイオメトリクス等）。認可（Authorization / AuthZ）: 認証済みユーザーが「何をして良いか？」を制御するプロセス。ロールや権限（Permission）に基づきリソースへのアクセスを管理する。
わかりやすく認証は「社員証を見せて入館する」行為。認可は「入館はできたが、役員フロアには立ち入り禁止」というアクセス制限。必ず認証→認可の順序で動作します。
具体例GitHubにログイン（認証）→ 自分のリポジトリは編集可・他人のリポジトリは読み取り専用（認可）

JWT（JSON Web Token）

認証済みユーザー情報をJSON形式でエンコードした自己完結型のトークン。ヘッダー・ペイロード・署名の3部構成でBase64URL形式の文字列になる。サーバーがセッションを保持しなくて良いためスケールしやすい。REST APIの認証（Bearer Token）として広く使われる。

OAuth 2.0（オーオース）

「Googleでログイン」などを実現する認可フレームワーク（RFC 6749）。サードパーティアプリが、ユーザーの代わりにリソースへ限定的にアクセスできる仕組み。よく見る「Googleアカウントで続ける」ボタンの裏側がOAuth 2.0です。認証のためにOIDC（OpenID Connect）が拡張として使われることも多い。

SSO（Single Sign-On・シングルサインオン）

一度のログインで複数のシステムを利用できる仕組み。企業では「社内ポータルにログインすれば、Slack・Jira・Confluenceも自動ログイン」のように使われる。IdP（Identity Provider）がOAuth 2.0やSAMLプロトコルで実現する。

セッションとCookie（従来型認証）

ログイン後サーバーがセッションIDを発行し、ブラウザのCookieに保存する。次回リクエストでCookieを送ることで認証状態を維持する。JWTと異なりサーバー側にセッション情報を保持する必要がある。スケールアウト時にセッション共有の仕組みが必要になる。

認証と認可の違いを教えてください
どちらもセキュリティに関する似たような概念だと思います。
認証（Authentication）は「あなたは誰か？」を確認するプロセスで、パスワードや生体認証で身元を証明します。認可（Authorization）は認証後に「あなたは何をして良いか？」を制御するプロセスで、ロールや権限に基づきリソースアクセスを管理します。GitHubを例にすると、ログインが認証で、「自分のリポジトリは書き込み可・他人のは読み取り専用」という権限制御が認可です。英語ではAuthN・AuthZと略されます。Web APIではJWT（JSON Web Token）でユーザー情報をトークン化して認証状態を管理し、Role-Based Access Control（RBAC）で認可を実装するのが一般的です。

📰 実際にあったできごと — SQLインジェクションによる大規模漏洩（2021年・国内ECサービス事例）

8-7 ゼロトラストセキュリティ（Zero Trust Security）

ゼロトラストセキュリティZero Trust Security / Zero Trust Architecture (ZTA)
正式定義「すべてのネットワーク・ユーザー・デバイスを信頼しない」という前提に基づくセキュリティモデル。2010年にJohn Kindervagが提唱。NIST SP 800-207で標準化。VPNによる「内側は安全」という境界防御モデルの限界を克服し、リソースへのすべてのアクセスを継続的に検証・最小権限で制御することを原則とする。
わかりやすく従来型は「社内ネットワークに繋がっていれば安全」（城壁モデル）。ゼロトラストは「社内でも社外でも、常に怪しいやつとして扱う」考え方。テレワーク普及・クラウド化で境界が曖昧になった現代のセキュリティの答え。
具体的な実装①ID基盤の強化（MFA必須、SSO、FIDO2）②デバイス信頼（MDM、EDR）③マイクロセグメンテーション ④継続的なアクセス監視（SIEM）⑤最小権限アクセス（PAM）
比較項目従来型（境界防御モデル）ゼロトラストモデル
基本前提内側（社内NW）は安全、外側は危険内外問わず全てのアクセスを検証
VPNの役割中心的なリモートアクセス手段VPNレス or ZTNA（Zero Trust Network Access）に移行
認証・認可ログイン一度で社内リソース全体にアクセス可リソースごとに継続的な認証・認可を実施
対応脅威外部からの侵入に強いが内部脅威に弱い内部不正・侵害済みアカウントにも有効
採用背景オンプレミス中心の時代クラウド化・テレワーク普及・内部脅威増加
ゼロトラストセキュリティとは何ですか？なぜ注目されているのですか？
社内でも外部でも安全を確認するセキュリティです。
ゼロトラストとは「誰も信頼しない」を原則とするセキュリティモデルです。従来の境界防御モデルは「社内ネットワークにいれば安全」という前提でしたが、クラウド化・テレワーク普及によって境界が曖昧になり、内部からの攻撃や侵害済みアカウントへの対処が困難になりました。ゼロトラストはリソースへのすべてのアクセスを継続的に検証し、最小権限でのみ許可します。具体的な実装としては多要素認証の徹底、デバイス管理（MDM）、マイクロセグメンテーション、SIEM（ログ監視）などが挙げられます。テレワーク常態化・ランサムウェア対策の観点から、企業のセキュリティ投資の最優先領域になっています。
💬 SQLインジェクションとは何ですか？どう防ぎますか？
友人説明入力欄に悪いSQLを書いてDBを操作する攻撃。プリペアドステートメントを使えばほぼ防げる。
面接基本SQLインジェクションはユーザー入力をSQL文に埋め込む際に悪意あるクエリが実行される脆弱性です。プリペアドステートメント（パラメータ化クエリ）による入力分離が基本的な対策です。
面接＋αORM使用でもraw queryには注意が必要です。WAFによる多層防御、最小権限原則（SELECT専用DBユーザー等）、定期的な脆弱性スキャン（sqlmap等）を組み合わせた多層防御が推奨されます。
模範回答私はセキュアコーディングレビューでSQLインジェクションリスクの洗い出しを担当し、全エンドポイントをパラメータ化クエリへ移行、OWASPのSQLi CheatSheetを基準にテストスイートを整備しました。脆弱性修正後はBurp Suiteで再検証するプロセスを確立しています。
💬 認証（Authentication）と認可（Authorization）の違いを教えてください
友人説明認証は「あなたは誰？」の確認、認可は「あなたは何をしていい？」の確認。ログインが認証、管理者画面アクセスが認可。
面接基本認証はユーザーのアイデンティティを確認するプロセス（ログイン）、認可は認証済みユーザーがどのリソースにアクセスできるか制御するプロセスです（権限管理）。
面接＋α認証方式はパスワード・MFA・パスキー、認可モデルはRBAC（ロールベース）・ABAC（属性ベース）があります。OAuthはアクセス委任の認可フレームワークで、OpenID Connect（OIDC）がその上に認証レイヤーを追加します。
模範回答私はマイクロサービス環境でJWT（RS256署名）による認証と、Open Policy Agent（OPA）によるポリシーアズコードの認可を分離実装しました。トークン短命化（15分TTL）+リフレッシュトークンローテーションによりセキュリティと利便性を両立しています。

9-1 ウォーターフォール開発

ウォーターフォールモデルWaterfall Development Model
正式定義要件定義→基本設計→詳細設計→実装→テスト→リリースというフェーズを順序通りに実行し、前フェーズの成果物を承認してから次フェーズに進む線形シーケンシャルな開発プロセスモデル。各フェーズに明確な完了定義があり、変更管理プロセスを通じてのみ前フェーズへの変更を許可する。SIerの大規模受託開発で広く採用される。
わかりやすく滝のように上から下へ一方通行で進む開発手法。先に全ての要件を確定してから作るため計画が立てやすい。しかし後から仕様変更があると手戻りコストが大きい。

9-2 アジャイル開発とスクラム（Agile / Scrum）

アジャイル開発 / スクラムAgile Development / Scrum Framework
正式定義アジャイル開発は、2001年に策定されたアジャイルマニフェストに基づく、反復的・漸進的なソフトウェア開発手法の総称。スクラムはアジャイルの具体的な実装フレームワークで、1〜4週間の固定サイクル（スプリント）でプロダクトバックログから作業を選択し、デイリースクラム・スプリントレビュー・レトロスペクティブという定例イベントを通じて自己組織化チームが継続的に改善する。
わかりやすく2週間ごとに動くソフトを作りながら、ユーザーの反応を見て方向修正していく手法。「全部作り終わってからリリース」ではなく、小さく作っては確認を繰り返す。自社サービス系企業やスタートアップで主流。

9-6 見積もり手法（Estimation Techniques）

プランニングポーカー / ストーリーポイントPlanning Poker / Story Points
正式定義プランニングポーカーは、開発チーム全員がフィボナッチ数列（1, 2, 3, 5, 8, 13, 21…）のカードを同時に出し合い、ストーリーポイントを合意形成するアジャイル見積もり手法。個人の見積もりを非公開で実施することで集団浅慮（アンカリング効果）を防ぎ、見積もりの根拠についての議論を促進する。
わかりやすく「このタスクは何時間かかる？」ではなく「他のタスクと比べてどれくらい大変か？」を数字にする手法。全員が同時に数字を出すことで、経験豊富なメンバーの意見に引きずられるのを防ぐ。
具体例ログイン機能を1（基準）とすると、検索機能は3、レコメンドエンジン連携は8、という相対見積もりをポーカーで合意する。
⚠️ 「ストーリーポイント＝時間」は誤り
ストーリーポイントは複雑度・不確実性・作業量を組み合わせた抽象指標です。「1ポイント＝1時間」と換算しないこと。ベロシティを通じて間接的に計画精度を上げるものです。
スクラムにおけるプロダクトオーナーとスクラムマスターの違いを教えてください
プロダクトオーナーがリーダーで、スクラムマスターはサポート役です。
プロダクトオーナー（PO）は「何を作るか」を決める役割で、プロダクトバックログを管理してビジネス価値を最大化する責任を持ちます。スクラムマスター（SM）は「スクラムを正しく実践できているか」を守護する役割で、チームの障害除去やデイリースクラムなどのイベントをファシリテーションします。POがビジネスサイドの代表とすれば、SMはプロセスの専門家です。どちらも開発チームに指示命令する立場ではない点がポイントです。
アジャイル開発での見積もり手法について教えてください
プランニングポーカーで時間を見積もります。
アジャイルでよく使われるのがプランニングポーカーによるストーリーポイント見積もりです。時間ではなく「複雑さ・不確実性・作業量」を相対評価で数値化します。全員が同時にフィボナッチ数列のカードを出すことで、アンカリング効果（先に聞いた数字に引きずられること）を防ぎ、見積もりの根拠についての議論を促します。チームのベロシティ（過去のスプリントの平均完了ポイント）を元に、次スプリントの計画量を決定します。

💬 Momoとめいめいの会話 — SpotifyはなぜAIが好みを分かるの？

🌸 Momo 天気予報に例えると分かりやすい！機械学習って「過去のデータからパターンを学習して予測する」技術なんですね。
💻 めいめい 完璧な理解です！「大量のデータから法則を見つけ、新しいデータに適用する」——それが機械学習の本質です。Spotifyは週1回、数億人分のデータを処理してプレイリストを更新しています。この規模は人間の手では不可能で、だからこそAI・MLエンジニアの需要が爆発しているんです。
人工知能（AI）とはAI: Artificial Intelligence
正式定義人間の知的行動（認識・推論・学習・判断）をコンピュータで模倣・代替する技術・システムの総称。1956年のダートマス会議でJohn McCarthyが提唱した概念。
わかりやすく「人間のように考えるコンピュータ」を作る技術の大きなカテゴリ。その中に機械学習・深層学習・生成AIが入れ子になっている。
具体例音声認識（Siri/Alexa）、画像認識（顔認証）、推薦システム（Netflix）、チャットボット（ChatGPT）

AI ⊃ Machine Learning ⊃ Deep Learning ⊃ Generative AI（入れ子構造）

機械学習（ML）Machine Learning
正式定義明示的にプログラムせずに、大量のデータからアルゴリズムが自動的にパターン・ルールを学習し、予測・判断を行う技術。Tom Mitchell（1997）の定義：「タスクTを経験Eから学習し、性能指標Pが向上するプログラム」。
わかりやすく「大量のお手本を見せて、機械自身にルールを覚えさせる」手法。人間がif-elseルールを書く代わりに、データから自動学習する。
種類教師あり学習（分類・回帰）、教師なし学習（クラスタリング）、強化学習（報酬最大化）
深層学習（DL）Deep Learning
正式定義人工ニューラルネットワーク（ANN）を多層に積み重ねた機械学習の一手法。入力層・隠れ層（複数）・出力層で構成され、各層が特徴を抽象的に表現する。画像・音声・自然言語処理で飛躍的な精度向上をもたらした。
わかりやすく人間の神経細胞（ニューロン）を模した多層ネットワーク。猫の写真を見て「耳の形→顔の輪郭→猫」と段階的に特徴を抽出する。
応用画像認識（ResNet）、翻訳（Transformer）、音声認識（Whisper）
大規模言語モデル（LLM）LLM: Large Language Model
正式定義数十億〜数千億のパラメータを持つTransformerアーキテクチャベースのニューラルネットワーク。大規模テキストコーパスで事前学習（pre-training）し、次トークン予測タスクによって言語の統計的パターンを学習する。代表例：GPT-4、Claude 3、Gemini Ultra、LLaMA 3。
わかりやすくインターネット上の膨大なテキストを読み込んだ「超巨大な予測機械」。「次に来そうな単語は何か？」を繰り返すことで、自然な文章を生成する。
具体例ChatGPT（GPT-4o）、Claude（Anthropic）、Gemini（Google）、Copilot（Microsoft）
RAG（検索拡張生成）RAG: Retrieval-Augmented Generation
正式定義LLMの推論時に外部知識ベース（ベクトルDB等）から関連文書を動的に検索し、その文書をコンテキストとして付与してから回答を生成するアーキテクチャ。Meta AIが2020年に提案。LLMの知識の鮮度問題・ハルシネーション問題を軽減する。
わかりやすくAIが回答する前に「参考書を調べてから答える」仕組み。社内マニュアルや最新情報をリアルタイムで参照できる。ChatGPTにWeb検索させる機能もこれに近い。
活用例社内FAQ チャットボット（就業規則を検索して回答）、法律相談AI（判例DBを参照）
ハルシネーションHallucination（幻覚）
正式定義LLMが事実と異なる情報を、あたかも正確であるかのように自信を持って生成する現象。訓練データの偏り、コンテキスト不足、統計的パターンへの過度な依存が原因とされる。
わかりやすくAIが「でたらめを自信満々に話す」こと。存在しない論文の引用、間違った法律の説明など。生成AIを業務利用する際は必ずファクトチェックが必要。
対策RAGによる根拠付き生成、Temperature値を低くする、Chain-of-Thought prompting

プロンプトエンジニアリング（Prompt Engineering）

プロンプトエンジニアリングPrompt Engineering
正式定義LLM（大規模言語モデル）から望ましい品質・形式の出力を引き出すために、入力プロンプト（指示文・コンテキスト・例示）を設計・最適化する技術。ゼロショット・フューショット・Chain-of-Thought（CoT）・ReActなど、複数のプロンプト戦略パターンが研究されている。
わかりやすくAIへの「質問・指示の仕方を工夫する技術」。同じAIでも、指示の書き方によって回答の質が大きく変わる。「要約して」より「200字以内で箇条書き3点に要約して」のほうが精度の高い出力が得られる。
主な手法①ゼロショット：例なしで直接指示 ②フューショット：例を2〜5個見せてから指示 ③Chain-of-Thought：「ステップバイステップで考えて」と論理過程を明示させる ④システムプロンプト：役割（ペルソナ）を事前に設定する
機械学習の「教師あり学習」と「教師なし学習」の違いを説明してください
ラベルがあるかないかの違いです。
教師あり学習は正解ラベルが付いたデータで学習し、新しいデータの予測や分類を行う手法です。スパムメール判定（スパム/正常）や住宅価格予測が代表例です。一方、教師なし学習は正解ラベルなしのデータからデータの構造やパターンを自律的に発見します。顧客を購買傾向でグループ化するクラスタリングや、異常な取引を検知する異常検知が代表例です。実務では教師あり学習が最も多く使われますが、ラベル付けコストが高い場合は教師なし学習や自己教師あり学習が有効です。
プロンプトエンジニアリングとは何ですか？なぜ重要ですか？
AIへの質問の仕方です。
プロンプトエンジニアリングとは、LLMから高品質な出力を引き出すために、入力テキスト（プロンプト）を設計・最適化する技術です。重要な理由は、同じLLMモデルでも指示の書き方によって出力品質が大幅に変わるためです。例えば単に「要約して」と指示するより「200字以内で箇条書き3点にまとめて」と指定するほうが使いやすい出力が得られます。Chain-of-Thoughtという手法では「ステップバイステップで考えて」と追記するだけで推論精度が向上することが研究で示されています。モデル開発ができない場面でも生産性を高められる実践的なスキルです。
💬 機械学習とは何ですか？ルールベースと何が違いますか？
友人説明ルールベースは人がif文を書く、機械学習はデータからコンピューターが自分でルールを見つける。
面接基本機械学習はデータから統計的パターンを学習しモデルを構築する技術です。ルールベースは人手でロジックを記述するのに対し、機械学習は大量データから汎化規則を自動抽出します。
面接＋α教師あり学習・教師なし学習・強化学習の3分類があります。過学習防止には正則化・ドロップアウト・早期停止を使用し、学習データの品質とバイアスがモデル性能を大きく左右します。
模範回答私は◯◯業務でXGBoostベースの異常検知モデルを実装し、Precision/Recall/F1を業務要件と照合しながら閾値チューニングを行いました。モデルの解釈可能性（SHAP）をステークホルダーへの説明に活用し、ブラックボックス問題を軽減しています。
💬 大規模言語モデル（LLM）について説明してください
友人説明GPTとかClaude的なやつ。大量のテキストで学習してて、次のトークンを予測することで文章を生成する。
面接基本LLMはTransformerアーキテクチャをベースに大規模テキストで事前学習された言語モデルです。文章生成・要約・翻訳・質疑応答など幅広いタスクをプロンプト指定でこなします。
面接＋α事前学習→RLHF（人間フィードバック強化学習）によるアライメントで有害出力を抑制します。RAGで外部知識を組み込み幻覚を低減、ファインチューニングで特定ドメインの性能を向上できます。

11開発プロセス・DevOps Development Process & DevOps

💬 Momoとめいめいの会話 — 「作る人」と「動かす人」はなぜ対立する？
🌸 Momo 「IT企業では開発チームと運用チームが仲悪い」ってよく聞くんですが、なんでそうなるんですか？
💻 めいめい それぞれの評価基準が真逆なんです。開発チームは「新機能を早くリリースした人が偉い」と評価される。でも運用チームは「サービスを止めなかった人が偉い」と評価される。新機能リリース＝コードの変更＝障害リスクなので、自然に対立が生まれます。レストランで言えば、料理長（開発）は「毎日新メニューを出したい」、ホールマネージャー（運用）は「今のメニューで手堅く回したい。変更は混乱のもと」という関係です。
🌸 Momo それを解決するのがDevOpsですか？でも「早くリリース」と「障害を出さない」って両立できるんですか？
💻 めいめい それを可能にするのが自動化です！コードをpushするたびに「自動テスト→ステージング確認→本番デプロイ」が走る仕組み（CI/CDパイプライン）を整備すれば、人の手を介さずに速さと品質を両立できます。AmazonはDevOps文化を徹底して、1時間に平均1,000回以上デプロイしながら99.99%以上の稼働率を維持しています。
DevOpsとはDevOps: Development + Operations
正式定義開発（Development）チームと運用（Operations）チームの協力・統合を促進し、ソフトウェアのデリバリーサイクルを高速化・高品質化するための文化・慣行・ツールセットの総称。Patrick DeboisとAndrew Claysが2009年に提唱。
わかりやすく従来「作る人」と「動かす人」は別チームで対立しがちだった。DevOpsはその壁を壊し、コードを書いてから本番に届けるまでのサイクルを自動化・高速化する考え方。
実現ツールGitHub Actions、Jenkins、Docker、Kubernetes、Terraform、Ansible

CI/CDパイプライン：コードから本番までを自動化するフロー

CI（継続的インテグレーション）CI: Continuous Integration
正式定義開発者が頻繁（1日複数回）にコードを共有リポジトリにマージし、その都度自動ビルドと自動テストを実行することで、統合作業の問題を早期に検出する開発手法。Martin Fowlerが体系化。
わかりやすくコードをpushするたびに「自動でビルドしてテストが走る」仕組み。バグを早期発見でき、「自分のコードが他のメンバーの動作を壊していないか」を即座に確認できる。
ツール例GitHub Actions、CircleCI、Jenkins、GitLab CI
CD（継続的デリバリー）CD: Continuous Delivery / Deployment
正式定義CIを経たコードをステージング環境・本番環境へ自動的にデプロイできる状態に常に保つ（Delivery）、あるいは自動的にデプロイする（Deployment）手法。リリース頻度を高め、手動ミスを削減する。
わかりやすく「いつでもワンクリックで（あるいは完全自動で）本番にリリースできる」状態を作ること。従来の「半年に1回の大リリース」から「1日数十回のリリース」を可能にする。
有名事例Amazon：1時間あたり平均1,000回以上のデプロイを実施

コードレビューのフロー

コードレビューCode Review
正式定義ソフトウェア開発において、作成者以外の開発者（レビュアー）がソースコードを精査し、バグ・設計上の問題・コーディング規約違反・セキュリティリスクを発見・修正する品質保証プロセス。プルリクエスト（PR）ベースのレビューが現代の標準。
わかりやすく「他の人に自分のコードを読んでもらい、問題がないか確認してもらう」作業。一人では気づかないバグや「もっと良い書き方」を指摘してもらえる。GitHubのPull RequestがこのUIを提供する。

環境管理：開発・ステージング・本番（Environment Management）

開発環境の種類Development / Staging / Production Environments
正式定義ソフトウェア開発では、コードの品質を守りながら安全にリリースするため、用途別に複数の「環境（Environment）」を分離して管理する。各環境はネットワーク・データベース・設定値がすべて独立しており、本番データへの意図しないアクセスや変更を防ぐ。
わかりやすく「書いたコードをいきなり本番サーバーで動かすのは危険すぎる」ので、開発→テスト→最終確認→本番の4段階に環境を分けて、段階的にリリースする仕組み。各段階でバグを発見・修正することで品質を積み上げる。
環境名英語名用途主な使用者
開発環境Development（dev）開発者が個人で実装・デバッグ。ローカルPCや個人用開発サーバー開発エンジニア

ステージング環境Staging（stg）本番と同じ構成でリリース前の最終確認。「本番の鏡」と呼ばれるQA・PO・ステークホルダー

コンテナ技術（Docker）

💬 Momoとめいめいの会話 — 「私のパソコンでは動くのに…」問題
🌸 Momo 先輩、「自分のPCでは動くのに本番サーバーでは動かない」ってよく聞くんですが、なんでそんなことが起きるんですか？
💻 めいめい あるある過ぎて泣けます（笑）。よくある原因は「Pythonのバージョンが開発PCは3.11なのに本番サーバーは3.8」「ライブラリのバージョンが違う」「MacとLinuxでOSの挙動が微妙に違う」といった環境の差です。英語では "It works on my machine" 問題と呼ばれています。
🌸 Momo じゃあ、みんなの環境を完全に同じにすることってできないんですか？
💻 めいめい それを解決したのが Docker です！アプリと「それが動くのに必要なすべての環境（OSのライブラリ・Pythonのバージョン・設定ファイルなど）」を丸ごと1つの箱（コンテナ）に詰め込めます。その箱ごとどこに持っていっても、まったく同じ環境で動きます。「環境ごとコピーして渡す」イメージです。
🌸 Momo 引っ越し業者が家具だけじゃなく「部屋の空気まで」丸ごと運んでくれるようなものですね！
💻 めいめい その例えすごく上手い！Dockerイメージ（箱の設計図）を1回作れば、開発PC・ステージング・本番サーバー・チームメンバーのPCすべてで完全に同じ環境が再現できます。だからCI/CDパイプラインとの相性も抜群で、今やほぼすべてのクラウドネイティブ開発で使われています。
コンテナ技術Container Technology（Docker / Kubernetes）
正式定義アプリケーションとその依存ライブラリ・設定をパッケージ化し、ホストOSのカーネルを共有しながら隔離された実行環境（コンテナ）を提供する技術。仮想マシン（VM）と比べてオーバーヘッドが少なく、秒単位で起動できる。Dockerが2013年に普及を推進し、現在はKubernetesによるコンテナオーケストレーションが大規模運用の主流。
わかりやすく仮想マシンが「PC丸ごとコピー」なのに対し、コンテナは「アプリに必要なものだけを小箱に詰めた」軽量版。同じDockerイメージから何十個ものコンテナを瞬時に起動でき、クラウドでの自動スケーリングと相性抜群。
主要ツールDocker（コンテナの作成・実行）、Docker Compose（複数コンテナ定義・管理）、Kubernetes / K8s（大規模コンテナオーケストレーション）、AWS ECS・GKE・Azure AKS（クラウドのマネージドコンテナサービス）
比較項目仮想マシン（VM）コンテナ（Docker）
起動時間数分数秒〜1秒以下
イメージサイズ数GB（OS丸ごと含む）数十〜数百MB
隔離レベル完全隔離（独自OSカーネル）プロセス隔離（ホストカーネル共有）
ポータビリティ重い・移行コスト高軽い・どこでも同じ動作
主な用途完全なOS分離が必要な場面マイクロサービス・CI/CD・クラウド
💬 CI/CDとは何ですか？なぜ重要ですか？
友人説明コードをpushしたら自動でテストして自動でデプロイしてくれる仕組み。手作業ミスが減って爆速でリリースできる。
面接基本CIは継続的インテグレーション（コードマージ時に自動ビルド・テスト）、CDは継続的デリバリー/デプロイ（テスト通過後の自動リリース）です。品質維持とリリース頻度の向上が主な目的です。
面接＋αGitHub ActionsやCircleCIでパイプラインを定義し、ユニット/統合/E2Eテストを段階的に実行します。ブルーグリーンデプロイやカナリアリリースで本番リスクを最小化し、失敗時の自動ロールバックも重要要素です。
模範回答私はGitHub Actionsで3ステージパイプライン（Lint→Test→Deploy）を構築し、PRマージからStaging展開まで8分→2分30秒に短縮しました。DORA指標（デプロイ頻度・変更リードタイム・障害復旧時間・変更失敗率）を追跡し、エリートパフォーマーレベルを達成しています。

💻 めいめい 昔はそうでしたよ！データセンターを借りて自前のサーバーを並べる「オンプレミス」という方式でした。でも今はクラウドが主流で、必要なサーバーをAWSやGCPから必要なぶんだけ借りてすぐ使えます。インフラ担当の仕事は「いかに障害を防ぎ・負荷を分散し・コストを最適化するか」という設計が核心になります。

ロードバランサーLoad Balancer（負荷分散装置）
正式定義複数のサーバーへトラフィックを分散させ、単一サーバーへの過負荷を防ぎ、可用性とスループットを向上させるネットワーク機器またはソフトウェア。分散アルゴリズムとしてラウンドロビン、最小接続数、IPハッシュ等がある。
わかりやすく「多くの客が来ても、複数のレジに振り分ける」係の仕事をするシステム。1台のサーバーがパンクしないよう、10台のサーバーに均等に仕事を割り振る。
AWSでの実装ALB（Application Load Balancer）：L7でHTTPパスやヘッダーで振り分け。NLB（Network Load Balancer）：L4でTCP/UDPを超高速処理。

マイクロサービスアーキテクチャ：各機能を独立したサービスとして分割・独立デプロイ可能

CDN（コンテンツ配信ネットワーク）CDN: Content Delivery Network
正式定義地理的に分散したエッジサーバー群を通じて、静的コンテンツ（HTML・CSS・画像・動画等）をユーザーの物理的に近いノードからキャッシュ配信する分散インフラ。オリジンサーバーへの負荷軽減・レイテンシ削減・可用性向上を実現する。
わかりやすく「東京のサーバーの内容を大阪・福岡・NY・Londonにも複製しておく」仕組み。ユーザーが大阪にいれば大阪のサーバーから届くので速い。
主要サービスCloudflare、AWS CloudFront、Akamai、Fastly
マイクロサービスMicroservices Architecture
正式定義アプリケーションを小さな独立したサービス群に分割し、各サービスが独自のプロセスで動作し、軽量なAPI（主にHTTP/REST またはメッセージキュー）で通信するソフトウェアアーキテクチャスタイル。モノリシックアーキテクチャの対概念。
わかりやすく「巨大なプログラムを機能ごとに分割して、別々にデプロイ・スケールできるようにする」設計。ユーザー管理・注文管理・決済を独立したサービスとして開発・運用。
採用事例Netflix、Amazon、Uber（各機能を独立サービスとして数百のマイクロサービスで構成）

💬 コンテナと仮想マシン（VM）の違いを教えてください

DNSとは何ですか？
「ドメインをIPに変換するサーバーです」
「DNS（Domain Name System）は人間が読めるドメイン名（例：google.com）を、コンピュータが通信に使うIPアドレス（例：142.250.196.14）に変換する分散型データベースシステムです。電話帳に例えられます。問い合わせはキャッシュDNS→ルートDNS→TLDサーバー→権威DNSの順で再帰的に行われます。DNSの障害はサイト全体のアクセス不能に繋がる重要なインフラです。」

🗄️ データベース・SQL系

トランザクションとは何ですか？なぜ重要ですか？
「データベースの操作の単位です」
「トランザクションとは、複数のDB操作を一つの論理的な作業単位としてまとめ、全て成功するか全て失敗するかを保証する仕組みです。銀行振込を例にすると、A口座から引き落とし→B口座へ入金の2操作は必ずセットで完了する必要があります。途中で失敗した場合はROLLBACKで元に戻します。ACID特性（Atomicity原子性・Consistency一貫性・Isolation独立性・Durability永続性）により、データの整合性を保証します。」
SQLのINNER JOINとLEFT JOINの違いは？
「JOINはテーブルを結合するもので、種類がいくつかあります」
「INNER JOINは両方のテーブルに一致するレコードのみを返します。LEFT JOINは左テーブルの全レコードを返し、右テーブルに一致がなければNULLを入れます。例えば受注テーブルと顧客テーブルをJOINする場合、INNER JOINは顧客情報のある受注のみ表示、LEFT JOINは顧客情報がない受注もNULLで含めて表示します。未登録顧客の注文を漏れなく拾いたいときはLEFT JOINを使います。」
RDBMSとNoSQLの使い分けを説明してください。
「NoSQLは新しい技術でスケールしやすいと聞いています」
「RDBMSはテーブル間の関係を定義し、SQL・ACID・JOIN等で複雑なデータ整合性を保証します。EC注文・銀行取引など整合性が最優先の用途に向いています。NoSQLはスキーマレスで水平スケールが容易で、大量の非構造化データや高スループット用途に向きます。例えば、ユーザーの購買履歴（MySQL）とセッション情報（Redis：KVS）、SNSのフィード（MongoDB：ドキュメント型）を用途別に使い分けることが現実的な設計です。」

☁️ クラウド・インフラ系

DockerとVMの違いは何ですか？
「Dockerはコンテナを使う技術で、VMより軽いと聞いています」
「仮想マシン（VM）はハイパーバイザーでホストOSの上に完全なゲストOSを仮想化するため、起動に数分かかりGBスケールのメモリを消費します。Dockerコンテナはホストカーネルを共有し、アプリとその依存ライブラリだけをパッケージ化するため、秒単位での起動とMBスケールの軽量性が特徴です。『私の環境では動く』という問題をDockerfileで依存関係を固定することで解消し、開発・テスト・本番で同一環境を再現できます。」
IaaS・PaaS・SaaSの違いを教えてください。
「クラウドサービスの種類で、SaaSが一番使いやすいものです」
「三つはクラウドサービスの責任範囲の違いです。IaaS（AWS EC2等）はインフラ（サーバー・ネットワーク）のみ提供し、OSからアプリまで利用者が管理します。PaaS（Heroku・AWS Elastic Beanstalk等）はOSやランタイムも提供し、利用者はアプリのコードに集中できます。SaaS（Salesforce・Gmail等）は全て提供済みでユーザーは機能を使うだけです。ITエンジニアとしてはIaaSとPaaSの境界を理解し、適切なサービスを選定することが求められます。」

🔒 セキュリティ系

SQLインジェクションとその対策を説明してください。
「不正なSQLを入力する攻撃で、対策が必要です」
「SQLインジェクションは、ユーザー入力値に悪意あるSQL文を埋め込み、DBを不正操作する攻撃です。例えばログインフォームに ' OR '1'='1 を入力してWHERE句を常にtrueにし、認証をバイパスします。対策の第一はプリペアドステートメント（パラメータ化クエリ）で、SQLと入力値を分離処理することで入力値がSQL構文として解釈されません。次にORMの利用、入力値のバリデーション、最小権限のDBユーザー設定を組み合わせます。」
XSSとCSRFの違いを説明してください。
「どちらもWebの攻撃で、セキュリティ対策が必要なものです」
「XSS（クロスサイトスクリプティング）は攻撃者が悪意のあるJavaScriptをWebページに埋め込み、他ユーザーのブラウザで実行させる攻撃です。被害者のCookieやセッション情報を盗めます。対策は出力時のHTMLエスケープとCSP（Content Security Policy）設定です。CSRF（クロスサイトリクエストフォージェリ）は被害者が意図しないリクエストを正規サービスへ送信させる攻撃です。対策はCSRFトークン（ランダムな秘密値をフォームに埋め込む）とSameSite Cookie属性です。」

💻 システム設計面接入門

システム設計面接（System Design Interview）はミドル〜シニアエンジニア採用で頻出の選考ステップです。文科系学生でも「思考の枠組み」を先に覚えておくと差別化できます。

🔷 4ステップ回答フレームワーク

① 要件の明確化（2-3分）

DAU × アクション数 = QPS。ストレージは1投稿100Byte × 日次書き込み数 × 保持年数で計算。
③ 高レベル設計（10-15分）
クライアント → CDN → LB → API層 → DB の基本構成図を描く。ボトルネックを声に出しながら進める。
④ 詳細設計（面接官が指定した箇所）
DBスキーマ・キャッシュ戦略・メッセージキュー・一貫性保証（Strong vs Eventual）を深掘り。

🔷 ケーススタディ：URL短縮サービス設計

要件例：DAU 100万、読み書き比10:1、URLの保持5年、P99レイテンシ <100ms
概算：書き込み100QPS / 読み込み1,000QPS。5年ストレージ ≈ 1億URL × 500Byte ≈ 50GB
